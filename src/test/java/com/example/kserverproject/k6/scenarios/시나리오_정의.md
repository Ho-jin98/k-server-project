# 🧪 K6 부하 테스트 시나리오 정의서

> **사전 점검 사항**
> 1. 애플리케이션이 `localhost:8080`에서 실행 중이어야 함
> 2. Redis, Kafka가 로컬에서 실행 중이어야 함
> 3. 메뉴가 최소 3개 이상 등록되어 있어야 함 (ADMIN 계정으로 사전 등록)
> 4. K6 설치 완료 (`brew install k6` 또는 `https://k6.io/docs/get-started/installation/`)

---

```bash
# 1. 스모크 테스트로 모든 API 정상 동작 확인
k6 run scripts/99-smoke.js

k6 run scripts/05-kafka-integrity.js
```

```bash
k6 run -e BASE_URL=http://localhost:8081 scripts/01-order-concurrency.js
```

## 🎯 시나리오별 상세

### 01. 주문 동시성 (Redisson 분산락)

| 항목 | 값 |
|---|---|
| 검증 대상 | `RedisLockService.executeWithLock` (Redisson RLock) |
| VU | 30 |
| 지속 시간 | ~30초 |
| 핵심 요구사항 | 다수 인스턴스에서 동일 유저 동시 주문 시 잔액 음수 방지 |

#### 해피 패스
- 잔액 9,000P, 가격 3,000P 메뉴 → 30 VU 동시 주문 → 정확히 3건만 성공

#### 검증 지표
- ✅ `order_success_count == 3`
- ✅ `order_failed_insufficient == 27`
- ✅ 최종 잔액 = 0 (음수 절대 X)
- ✅ p95 응답 시간 < 5초 (락 대기 포함)

#### 실패 시 의심 포인트
- 잔액 음수 → 분산락이 동작 안 함 또는 트랜잭션 분리 문제
- 모두 성공 → 락 자체가 안 잡힘
- 너무 많은 503/500 → 락 대기 타임아웃 설정 문제

### 02. 포인트 충전 동시성 (DB 비관락)

| 항목 | 값 |
|---|---|
| 검증 대상 | JPA `@Lock(PESSIMISTIC_WRITE)` |
| VU | 20 |
| 지속 시간 | ~20초 |
| 핵심 요구사항 | 동시 충전 시 Lost Update 방지 |

#### 해피 패스
- 1유저, 20 VU가 각자 1,000P 충전 → 최종 잔액 = 20,000P

#### 검증 지표
- ✅ `charge_success_count == 20`
- ✅ 최종 잔액 = 시작 잔액 + 20,000P
- ✅ PointHistory의 balanceAfter 누적 일관성

#### 실패 시 의심 포인트
- 잔액 < 20,000P → 비관락 미적용 또는 트랜잭션 격리 부족
- 데드락 발생 → 락 획득 순서 문제

### 03. 메뉴 캐시 부하 (@Cacheable)

| 항목 | 값 |
|---|---|
| 검증 대상 | `@Cacheable` (Redis 30분 TTL) |
| VU | 50 → 100 → 200 (ramp) |
| 지속 시간 | ~1분 30초 |
| 핵심 요구사항 | 캐시 hit 시 빠른 응답, 부하에도 일관된 성능 |

#### 해피 패스
- Cold start 1회 → Warm 상태로 200 VU까지 ramp
- 메뉴 목록/상세/검색 3가지 패턴 랜덤 호출

#### 검증 지표
- ✅ `menu_list_duration p95 < 200ms`
- ✅ `cache_hit_likely > cache_miss_likely` (대부분 캐시 hit)
- ✅ 실패율 < 1%

#### 실패 시 의심 포인트
- p95가 너무 큼 → 캐시 미적용, 매번 DB 조회
- cache_miss_likely가 많음 → TTL 너무 짧거나 캐시 키 불일치

#### 추가 검증 (수동)
```bash
# Redis CLI에서 캐시 키 확인
redis-cli KEYS "menus*"
redis-cli TTL "menus::SimpleKey []"
```

### 04. 인기메뉴 조회 (Sorted Set + result 키 캐싱)

| 항목 | 값 |
|---|---|
| 검증 대상 | `popular:menus:result` 캐싱 + ZUNIONSTORE |
| VU | 100 |
| 지속 시간 | 1분 |
| 핵심 요구사항 | 부하가 몰려도 ZUNIONSTORE 최대 1번만 실행 |

#### 해피 패스
- setup에서 시드 주문 → 5초 대기 → 베이스라인 조회
- 100 VU 동시 조회 → 모든 응답이 동일한 TOP 3

#### 검증 지표
- ✅ `popular_consistency_match >> mismatch`
- ✅ p95 응답 시간 < 100ms (result 키 캐시 적중)
- ✅ 모든 응답 200 OK

#### 실패 시 의심 포인트
- p95가 큼 → result 키 캐싱 미동작, 매 요청마다 ZUNIONSTORE
- 응답 불일치 → result 키 갱신 중 동시 요청에서 일관성 깨짐

#### 추가 검증 (수동)
```bash
# 날짜별 키 확인
redis-cli KEYS "popular:menus:*"

# 오늘 날짜 메뉴별 카운트
redis-cli ZREVRANGE popular:menus:2026-05-07 0 -1 WITHSCORES

# result 키 TTL 확인
redis-cli TTL popular:menus:result
```

### 05. Kafka 정합성 (가장 중요)

| 항목 | 값 |
|---|---|
| 검증 대상 | Producer → Consumer → ZINCRBY (메시지 유실 방지) |
| VU | 20 |
| Iteration | 50건 |
| 핵심 요구사항 | **메뉴별 주문 횟수가 정확해야 함** |

#### 해피 패스
- 베이스라인 카운트 조회
- 50건 주문 발행 → 10초 대기 → ZSCORE 검증
- 카운트 증가량 = 50 (정확히)

#### 검증 지표
- ✅ `order_published == 50`
- ✅ `order_failed == 0`
- ✅ 최종 ZSCORE = 베이스라인 + 50

#### 실패 시 의심 포인트
- 카운트가 < 50 → Kafka 메시지 유실 (DLT 확인 필요)
- 카운트가 > 50 → Consumer 중복 처리 (멱등성 문제)

#### ⚠️ 주의: result 키 캐싱 영향
- 인기메뉴 API는 result 키 (TTL 1시간) 때문에 즉시 반영 안 될 수 있음
- 정확한 검증은 **Redis CLI로 직접** ZSCORE 확인이 필수
```bash
redis-cli ZSCORE popular:menus:$(date +%Y-%m-%d) {menuId}
```

#### Kafka 측 검증 (수동)
```bash
# 토픽 확인
kafka-topics.sh --bootstrap-server localhost:9092 --list

# DLT 메시지 확인 (있으면 안 됨)
kafka-console-consumer.sh --bootstrap-server localhost:9092 \
  --topic order-event.DLT --from-beginning --timeout-ms 5000

# Consumer 그룹 lag 확인 (0이어야 함)
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group order-consumer-group
```

## 🛠️ 트러블슈팅

### "메뉴가 없습니다" 에러
ADMIN 계정으로 메뉴 3개 이상 등록 필요. SQL이나 별도 스크립트로 시드 데이터 삽입.

### Kafka 정합성 테스트에서 카운트 차이 발생
1. result 키 TTL 때문일 수 있음 → Redis CLI로 직접 ZSCORE 조회
2. Consumer가 죽었는지 확인 → `kafka-consumer-groups.sh --describe`
3. DLT 토픽에 메시지 쌓였는지 확인

### 동시성 테스트에서 잔액이 어중간하게 남음
- 락 대기 시간(waitTime: 3초)이 짧아서 일부 요청이 락 획득 실패한 경우
- VU 수를 줄이거나 waitTime을 늘려서 재실행
